# gc-lang v1.0.0 — API Freeze

**The stable release.** v0.2.0 delivered the collector; v1.0.0 freezes it. The public
surface is now locked for the entire `1.x` series — no breaking changes until `2.0` —
and the crate is production-ready. There are no functional changes since 0.2.0: this
release is the commitment, not new code.

## What is gc-lang?

A tracing, cycle-collecting garbage collector for interpreted-language runtimes, in
safe Rust (`#![forbid(unsafe_code)]`, `no_std` on `alloc`, zero runtime dependencies).
Allocate objects into a `Heap<T>`, refer to them by `Gc<T>` handle, wire object graphs
by handle, and reclaim everything unreachable — cycles included — with `collect`.

## The frozen surface

Six items, stable for `1.x`:

| Item | Role |
|------|------|
| `Heap<T>` | the garbage-collected object store: `new`, `with_capacity`, `alloc`, `try_alloc`, `get`, `get_mut`, `contains`, `len`, `is_empty`, `capacity`, `collect` |
| `Gc<T>` | an 8-byte, `Copy`, generation-stamped handle; `Eq`/`Ord`/`Hash` for any `T` |
| `Trace` / `Tracer` | the reachability contract — an object reports the handles it owns |
| `CollectStats` | the `live` / `freed` result of a collection (`#[non_exhaustive]`) |
| `GcError` | the fallible-allocation error (`#[non_exhaustive]`) |

Every public method, signature, and documented behaviour on the
[API reference](https://github.com/jamesgober/gc-lang/blob/main/docs/API.md) is covered.

## The SemVer promise

For the whole `1.x` series:

- **No breaking changes.** No exported item is removed or renamed; no signature, bound,
  or documented behaviour changes incompatibly. Anything that would break a downstream
  crate is a `2.0`.
- **Additive minors.** New methods, types, and trait impls may arrive in a `1.y`
  release. The two `#[non_exhaustive]` types (`CollectStats`, `GcError`) may gain fields
  or variants — they are already marked so a downstream `match` accounts for it.
- **Patches** carry fixes, docs, and performance work with no surface change.

Not covered: exact `Debug` wording, precise benchmark figures, internal slot layout, and
the specific generation value a reused slot carries (only the guarantee that a stale
handle resolves to `None`). MSRV increases are at least a minor bump and recorded in the
CHANGELOG.

## Why freeze a minimal surface

The collector does one thing — allocate, resolve, and reclaim by reachability — and its
surface is deliberately small. Freezing it now, rather than accreting speculative
configuration first, is the point: downstream runtimes get a contract they can build on,
and everything genuinely useful later (`Heap::clear`, live-object iteration, incremental
collection modes, byte accounting) is **additive** and ships in `1.x` without breaking
anyone. See `dev/ROADMAP.md` → *Beyond 1.0*.

## Breaking changes

**None.** The 0.2.0 surface is promoted to stable unchanged.

## What's next

`1.x`, additive only. None of these is required, and none will break the frozen surface:

- `Heap::clear` — drop every object and reset the heap in one call.
- Live-object iteration for debugging and snapshotting.
- Incremental / generational collection modes for larger heaps.
- Per-collection byte accounting in `CollectStats`.

## Installation

```toml
[dependencies]
gc-lang = "1.0"

# no_std (needs a global allocator):
gc-lang = { version = "1.0", default-features = false }
```

MSRV: Rust 1.85 (Rust 2024 edition).
